Data Preparation

This page documents how the VAST Challenge 2026 Mini-Challenge 1 data, stored as a nested RDS object, is converted into clean, reusable tables for the investigation and the Shiny application. The RDS object was previously created from the supplied JSON data. This page focuses only on data loading, validation and feature engineering; no charts are produced here.

1 Setup

1.2 Loading the raw MC1 data

The supplied Mini-Challenge 1 dataset has already been converted from JSON into an RDS object named mc1_data.rds. It contains a sequence of simulation rounds, each with participants, communication records and environmental context. The RDS object is loaded using a relative path so that the project can be rendered on another group member’s computer without changing the code.

Code 
# This QMD file is stored in the Methodology folder.
# Therefore, the source data is referenced relative to this folder.
data_path <- file.path("data", "MC1", "mc1_data.rds")

if (!file.exists(data_path)) {
  stop(
    "Dataset not found. Please save mc1_data.rds in: ",
    normalizePath(file.path("data", "MC1"), winslash = "/", mustWork = FALSE)
  )
}

# Load the nested RDS object that was previously created from the MC1 JSON file.
mc1_rds <- readr::read_rds(data_path)

if (!is.list(mc1_rds) || is.null(mc1_rds$rounds)) {
  stop(
    "mc1_data.rds was loaded, but it does not contain the expected $rounds element. ",
    "Please ensure that this is the MC1 RDS file created from MC1_final_00.json."
  )
}

# The source RDS stores deeply nested lists. This standardisation step keeps the
# existing unnesting code below compatible with the RDS version of the data.

4 Communication feature engineering

The following features prepare the message-level data for later investigation of event sequence, communication exposure, behavioural signals and possible dissemination pathways. These are screening features only; they do not by themselves establish intent or wrongdoing.

4.1 Temporal features

The inappropriate release occurred on 5 June 2046. The features below distinguish normal activity from the escalation and critical windows immediately before the breach.

Code 
breach_date <- lubridate::ymd("2046-06-05")

communications_clean <- communications %>%
  mutate(
    timestamp = lubridate::ymd_hms(timestamp, quiet = TRUE),
    round_hour = lubridate::ymd_hms(round_hour, quiet = TRUE)
  ) %>%
  arrange(timestamp) %>%
  mutate(
    communication_date = as_date(timestamp),
    communication_hour = hour(timestamp),
    communication_day = wday(timestamp, label = TRUE),

    communication_period = case_when(
      communication_hour >= 6 & communication_hour < 12 ~ "Morning",
      communication_hour >= 12 & communication_hour < 18 ~ "Afternoon",
      communication_hour >= 18 & communication_hour < 24 ~ "Evening",
      TRUE ~ "Late Night"
    ),

    days_before_embargo_breach = as.integer(breach_date - communication_date),

    near_breach_window = case_when(
      days_before_embargo_breach < 0 ~ "Post-breach",
      days_before_embargo_breach <= 2 ~ "Critical Period",
      days_before_embargo_breach <= 5 ~ "Escalation Period",
      TRUE ~ "Normal Period"
    ),

    message_sequence = row_number()
  )

4.2 Communication scope and exposure risk

Channels are grouped into broader communication scopes. This allows later analyses to distinguish public communication from private, internal and side-channel exchanges.

communications_clean <- communications_clean %>%
  mutate(
    communication_scope = case_when(
      channel %in% c("official_post", "personal_post") ~ "Public",
      channel %in% c("one_on_one_chat") ~ "Private",
      channel %in% c("side_huddle") ~ "Side-channel",
      channel %in% c("comms_huddle") ~ "Internal Group",

      communication_scope == "Public" ~ "High Exposure",

      communication_scope == "Private" ~ "Low Exposure",

4.3 Text and behavioural indicators

Message content and any available internal-state text are combined into one analysis field. Keyword indicators then identify potential references to the merger, embargo controls, public disclosure and operational pressure.

Code 
combine_text_fields <- function(data, columns) {
  if (length(columns) == 0) {
    return(rep("", nrow(data)))
  }

  apply(data[columns], 1, function(x) {
    stringr::str_squish(
      stringr::str_c(replace_na(as.character(x), ""), collapse = " ")
    )
  })
}

internal_state_columns <- names(communications_clean) %>%
  stringr::str_subset("^internal_state_")

communications_clean$internal_state_text <- combine_text_fields(
  communications_clean,
  internal_state_columns
)

communications_clean <- communications_clean %>%
  mutate(
    content_lower = str_to_lower(replace_na(content, "")),
    analysis_text = str_squish(
      str_c(content_lower, str_to_lower(internal_state_text), sep = " ")
    ),

    contains_project_terms = str_detect(
      analysis_text,
      "harborcrest|civicloom|merger|rebrand|capital|deal"
    ),

    contains_embargo_terms = str_detect(
      analysis_text,
      "embargo|judge|compliance|approved|blocked|deadline"
    ),

    contains_public_exposure_terms = str_detect(
      analysis_text,
      "public|post|fle?x|release|leak|disclose|announcement"
    ),

    contains_pressure_terms = str_detect(
      analysis_text,
      "pressure|urgent|concern|risk|legal|scrutiny|journal|algorithmiceviction"
    ),

    has_internal_reasoning = str_length(internal_state_text) > 0,
    message_length = str_length(content_lower),
    word_count = str_count(content_lower, "\\S+"),

    risk_signal_score =
      as.integer(contains_project_terms) +
      as.integer(contains_embargo_terms) +
      as.integer(contains_public_exposure_terms) +
      as.integer(contains_pressure_terms),

    risk_signal_group = case_when(
      risk_signal_score >= 3 ~ "High Signal",
      risk_signal_score == 2 ~ "Moderate Signal",
      risk_signal_score == 1 ~ "Low Signal",
      TRUE ~ "No Signal"
    )
  )

risk_signal_summary <- communications_clean %>%
  count(near_breach_window, risk_signal_group, sort = TRUE)

5.2 Nodes and weighted edges

Nodes represent all unique senders and recipients. Weighted edges aggregate repeated sender-recipient interactions and retain communication scope and risk classification.

Code 
communication_nodes <- tibble(
  name = sort(unique(c(communication_edges$from, communication_edges$to)))
) %>%
  left_join(
    communications_clean %>%
      distinct(agent_id, agent_role) %>%
      rename(name = agent_id, role = agent_role),
    by = "name"
  ) %>%
  mutate(role = replace_na(role, "Recipient / System"))

communication_edges_weighted <- communication_edges %>%
  count(
    from,
    to,
    communication_scope,
    exposure_risk_level,
    risk_signal_group,
    name = "weight"
  ) %>%
  mutate(
    interaction_intensity = case_when(
      weight >= 10 ~ "High Interaction",
      weight >= 5 ~ "Moderate Interaction",
      TRUE ~ "Low Interaction"
    )
  )

    "Directed communication edges",
    "Weighted communication edges"
  ),
  Value = c(
    nrow(communication_nodes),
    nrow(communication_edges),
    nrow(communication_edges_weighted)
  )
)

6 Saving prepared data

All cleaned tables and engineered features are saved as one RDS object. The other Quarto pages and the Shiny application can load this prepared file directly rather than repeatedly unpacking the nested source RDS object.

Code 
prepared_mc1 <- list(
